# Third-Party JavaScript Security Demo

CSP と SRI を用いた第三者 JavaScript 改ざん対策の検証デモ

## 1. 概要

本プロジェクトは、第三者 JavaScript を利用する一般的な Web サイトを題材に、外部スクリプトの改ざんや差し替えによってどのような危険が生じるのか、またそれに対して CSP と SRI を用いることでどこまで対策できるのかを確認するためのデモ環境である。

近年、外部 JavaScript を悪用した攻撃や、正規サイト上で不正な JavaScript が実行される事例が報告されている。こうした背景を受け、本プロジェクトでは「第三者 JavaScript を読み込むサイト」において、攻撃が成立する条件と、防御策の役割分担を理解することを目的として実装を進めている。

本デモは、就職活動において以下を示すことを意図している。
- Web における第三者 JavaScript 利用の危険性を理解していること
- 単なる概念理解にとどまらず、実際に再現環境を構築して検証していること
- CSP と SRI の違いと役割を整理して説明できること
- 攻撃シナリオを前提に、防御策の設計と評価を行おうとしていること

## 2. このプロジェクトに取り組んだ背景

外部 JavaScript は、推薦表示、キャンペーン表示、分析タグなど、多くの Web サイトで一般的に利用されている一方で、こうしたスクリプトは自サイトが直接管理していない場合も多く、配信元の侵害や読み込み先の差し替えが起きた場合、利用者のブラウザ上で不正な処理が実行される危険性がある。

本プロジェクトでは、こうした「第三者 JavaScript を信頼して読み込むことによる危険性」に着目した。
特に、外部スクリプトの改ざんや差し替えによって、画面の書き換えや偽の再認証画面の表示などが起こり得る点に関心を持ち、CSP と SRI による防御の有効性を確認する方針で進めている。

## 3. 想定している攻撃シナリオ

現在は、攻撃シナリオを 1 つに絞って具体化している。

### 攻撃シナリオ

通常のニュースサイトにログイン機能を持たせた上で、第三者 JavaScript が改ざんされた場合に、画面を書き換えて偽の再認証画面を表示し、利用者に再入力を促す。

### 狙い
- 見た目として自然な攻撃にする
- 単なるバナー改ざんではなく、利用者の入力を誘導する危険性を見せる
- 正規サイトに見えるが、外部スクリプトが悪質である状況を再現する

### 現時点の整理

当初はインライン実行による偽画面表示も検討したが、第三者 JavaScript の攻撃として見せるため、現在は「外部スクリプトが悪質だった場合」の再現を主軸としている。

## 4. ディレクトリ構成

```
project/
├── app/
│   ├── index.html
│   ├── style.css
│   └── app.js
├── vendor/
│   ├── external-recommendation.js
│   └── campaign-widget.js
└── attacker/
    ├── external-recommendation.js
    └── campaign-widget.js
```
**app/**

メインサイト本体を構成するディレクトリ。
ニュース一覧画面、ログイン UI、スタイル、サイト本体の挙動を管理する。

**vendor/**

正規の第三者 JavaScript を配置するディレクトリ。
本来読み込まれる外部部品を置いており、推薦表示やキャンペーン表示などの役割を持つ。

**attacker/**

攻撃用 JavaScript を配置するディレクトリ。
vendor/ にある正規ファイルと同名の攻撃版ファイルを用意し、配信元の差し替えや改ざん後の挙動を再現するために使う。

## 5. 各ファイルの役割

**app/index.html**

メインサイトの HTML。
ニュースサイト風の画面を持ち、検索欄、ニュース一覧、ログイン UI、会員向け表示などがある。
また、vendor/ 側の外部 JavaScript を読み込むための script タグがある。

**app/style.css**

メインサイトの見た目を整える CSS。
ニュースサイト風の UI、ログインモーダル、会員状態表示などの外観を担当。

**app/app.js**

サイト本体の JavaScript。
記事データの表示、検索処理、テーマ切り替え、簡易ログイン状態の切り替え、プロフィール表示などを担当。
現時点ではブラウザ内で状態を持つだけであり、データベースや本物の認証処理は使っていない。

**vendor/external-recommendation.js**

正規の推薦表示ロジック。
ニュース一覧の並び替えや推薦表示など、外部の便利機能を想定した部品を構成。

**vendor/campaign-widget.js**

正規のキャンペーン表示ロジック。
ページ内にキャンペーンや通知のような部品を追加する外部ウィジェットを想定。

**attacker/external-recommendation.js**

攻撃版の推薦スクリプト。
本来の推薦表示ではなく、表示内容の改ざんや、見えているページ情報の読み取り風デモなどを行う。

**attacker/campaign-widget.js**

攻撃版のキャンペーンスクリプト。
本来のキャンペーン表示の代わりに、偽の再認証 UI を表示し、利用者の入力を誘導するデモを行う。

## 6. 実行方法

メインサイト起動

```
cd project/app
python3 -m http.server 3000
```

正規ベンダーファイル配信
```
cd project/vendor

```

http://localhost:3000/index.html

## 8. 現在の理解

### CSP の役割

CSP は、許可していない配信元からスクリプトを読み込ませないための仕組み。
たとえば attacker を http://localhost:5000 に置き、CSP で http://localhost:4000 のみを許可すれば、5000 番からの差し替えは防げる。

### SRI の役割

SRI は、許可している配信元から取得したファイルであっても、中身が想定と異なれば実行させないための仕組み。
そのため、正規の配信元そのものが侵害された場合でも、ハッシュが一致しなければ防げる。

### 現時点での役割分担

- attacker が別配信元にある場合 → CSP が有効
- attacker が正規配信元そのものを侵害している場合 → CSP だけでは不十分で、SRI が必要

## 9. 現在残っている課題

### 9.1 攻撃シナリオの具体化

現時点では「偽の再認証画面を出す」攻撃を中心に整理していますが、まだ攻撃導線や見せ方をさらに自然にする余地がある。
ex:通常ログインの流れと再認証要求の見せ方の自然化

### 9.2 CSP の導入と実証

現時点では SRI の確認が先行しており、CSP の実証は十分ではない。
今後は attacker 側を別ポートで配信し、許可していない配信元からの読み込みを CSP でどのように止められるかを確認する必要がある。

### 9.3 攻撃の種類ごとの整理

現在の主軸は「第三者 JavaScript 改ざん」ですが、以下のように整理し直す必要がある。
- 正規配信元の中身改ざん
- 読み込み先 URL の差し替え
- ページ内での直接実行

その上で、各攻撃に対して CSP と SRI のどちらが有効かを整理。
